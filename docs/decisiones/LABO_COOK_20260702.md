## Problemas reales atacados

En el laboratorio el robot se trababa ante una silla/patas:

- avanzaba hacia el obstaculo;
- detectaba algo adelante;
- retrocedia;
- dejaba de ver las patas;
- volvia a intentar el mismo paso;
- repetia el forward/back durante varios segundos antes de destrabarse.

La causa probable es que las patas aparecian como obstaculos finos o ruido
intermitente: al retroceder, el LIDAR dejaba de verlas y el plan global seguia
queriendo pasar por el mismo lugar.

## Scripts nuevos

- `scripts/lab_record_all.sh`: crea `results/labo_demo/<timestamp>/`, graba
  rosbag, corre logger vivo, guarda metadata y genera reporte al cortar.
- `scripts/lab_live_logger.py`: registra `events.jsonl`, poses, goals, estados,
  cmd_vel, detecciones, particlecloud, ultimo path y `summary.json`.
- `scripts/lab_make_report.py`: genera `summary.md`, `timeline.csv` y
  `map_overlay.png`; si no hay `matplotlib`, cae a `map_overlay.svg`.
- `scripts/lab_record_rviz.sh`: grabacion opcional de pantalla con `ffmpeg`; si
  no existe `ffmpeg` o no hay `DISPLAY`, avisa y sale sin romper.
- `scripts/smoke_chair_obstacle.sh`: harness sintetico para validar que un
  obstaculo frontal persistente no produzca loop forward/back indefinido.

Scripts ajustados:

  elegir overlay con `WORKSPACE_SETUP`.

## Como grabar evidencia manana

Primero conectar al robot y prender sensores:

```bash
turtlebot_mode 0
ros2 service call /tb4_0/start_motor std_srvs/srv/Empty {}
ros2 service call /tb4_0/oakd/start_camera std_srvs/srv/Empty {}
ros2 topic hz /tb4_0/scan
```

Terminal 1 - evidencia:

```bash
./scripts/lab_record_all.sh tb4_0
```

Terminal 2 - navegacion:

```bash
ros2 launch maze_nav nav_tb4_live.launch.py \
  map_yaml:=maps/laberinto_lab_20260702.yaml ns:=tb4_0
```

Terminal 3 - mision:

```bash
ros2 launch maze_mission mission.launch.py \
  params_file:=$(pwd)/config/parte_c/real.yaml
```

Terminal 4 - detector:

```bash
ros2 launch maze_perception cone_detector.launch.py \
  params_file:=$(pwd)/config/parte_c/real.yaml
```

Terminal 5 - RViz:

```bash
rviz2 -d src/maze_nav/rviz/nav.rviz --ros-args -p use_sim_time:=false \
  --remap /tf:=/tb4_0/tf \
  --remap /tf_static:=/tb4_0/tf_static \
  --remap /scan:=/tb4_0/scan
```

Opcional, Terminal 6 - pantalla RViz:

```bash
./scripts/lab_record_rviz.sh results/labo_demo/<timestamp>
```

## Como postprocesar evidencia

Al cortar `lab_record_all.sh` con `Ctrl-C`, intenta correr automaticamente:

```bash
python3 scripts/lab_make_report.py results/labo_demo/<timestamp>
```

El directorio queda con:

- `rosbag/`: datos ROS crudos;
- `events.jsonl`: eventos estructurados;
- `poses.csv`, `goals.csv`, `states.csv`, `cmd_vel.csv`;
- `latest_path.csv` y `path_updates.csv`;
- `summary.json` y `summary.md`;
- `timeline.csv`;
- `map_overlay.png` o `map_overlay.svg`.

## Cambio de navegacion ante silla/obstaculo

En `maze_nav/navigator.py` se mantuvo la prioridad de seguridad:

- no se subieron velocidades;
- no se bajo `safety_stop`;
- los obstaculos dinamicos se mantienen durante el goal actual y solo se limpian
  al recibir un goal nuevo;
- ante clearance frontal menor que `safety_stop`, el robot entra a `RECOVERY` y
  frena, en vez de seguir avanzando si no logro registrar celdas nuevas;
- `RECOVERY` ahora tiene hold + backoff corto temporizado;
- si el obstaculo fino no queda bien registrado y el bloqueo frontal se repite,
  se marca una barrera dinamica chica delante del robot para forzar replanning;
- si tras `max_recovery_attempts` sigue bloqueado, publica `IDLE`, deja
  `cmd_vel=0` y reporta `blocked_max_recovery_attempts` en `/nav_debug`;
- si no hay plan de recuperacion, queda detenido con
  `blocked_no_recovery_plan`.

`/nav_debug` publica `std_msgs/String` con JSON:

- `state`, `reason`, `pose`, `goal`;
- `forward_clearance_m`, `safety_stop_m`;
- `recovery_attempts`, `max_recovery_attempts`, `front_blocked_hits`;
- `dyn_obstacle_cells`, `last_dyn_added`;
- `path_len`, `path_idx`, `last_plan_length_m`;
- `cmd.linear_x`, `cmd.angular_z`;
- `blocked_reason`.
